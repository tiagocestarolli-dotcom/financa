# Notas para um caminho comercial futuro

> Documento vivo. **Não afeta o desenvolvimento atual.** Atualizado conforme decisões relevantes vão sendo tomadas durante a construção do Finança.

---

## Propósito

Várias escolhas do projeto funcionam ótimo pra um usuário (Tiago) ou uma família, mas precisariam ser repensadas pra abrir o app comercialmente. Este arquivo registra cada uma dessas escolhas para revisita futura, sem desviar foco do trabalho atual.

---

## Decisões que precisam revisita pra escala comercial

### 1. Arquitetura single-file
**Hoje:** `index.html` único com ~10000 linhas (CSS + HTML + JS). Funciona bem porque é uma pessoa editando, deploy simples via GitHub Action.
**Comercial:** Inviável com equipe de mais de 1 dev. Migração pra build process (Vite/Webpack), componentização (React/Vue), code-splitting. Quebra do single-file também permite testes unitários por componente.

### 2. Schema "família única"
**Hoje:** State com keys hardcoded: `casa`, `marina`, `heitor`, `juliana`, `tiago`. Tiago é isolado por design (`isolado: true`).
**Comercial:** Conceito de família é forte e diferenciador, mas precisa virar configurável. Cada workspace teria N membros (sem nomes hardcoded), com tipos: titular / dependente / isolado / cônjuge. Subcategoria "exclusiva por categoria" generaliza bem.

### 3. Firestore como source-of-truth
**Hoje:** Documento `users/{uid}` com estado inteiro embutido. Realtime sync, persistência offline, simples.
**Comercial:** Custo escala mal (cada read/write é cobrado). Estado inteiro no doc limita a ~1MB por usuário, pouco com 5+ anos de gastos. Migração natural: Supabase ou PostgreSQL, com sync de realtime via solução separada.

### 4. Auth limitada a Google
**Hoje:** Só sign-in com Google.
**Comercial:** Email/senha, Apple Sign-In (obrigatório se for pra App Store iOS), recuperação de conta.

### 5. Multi-tenant não existe
**Hoje:** Cada usuário Firebase tem seu próprio documento isolado. Etapa B do roadmap (Juliana como segunda usuária) endereça parcialmente.
**Comercial:** Workspaces, convites, papéis (admin/membro/leitor), faturamento por workspace ou por usuário. Decisão estrutural cedo — repensar antes de virar débito técnico.

### 6. Onboarding zero
**Hoje:** App assume que usuário entende "gasto fixo", "subcategoria exclusiva", "grupo cross-cutting", "categoria isolada". Faz sentido pra Tiago, que desenhou.
**Comercial:** Tutorial primeiro-uso, dicas contextuais, defaults inteligentes (talvez detecção de banco do email pra pré-cadastrar contas), copy descomplicado.

### 7. Decisões opinionadas de produto
**Hoje:** Tiago é categoria isolada. Itaú default. Investimentos têm subcategoria fixa. Decisões assumem o perfil do Tiago.
**Comercial:** Tudo configurável via wizard de setup ("Quer uma categoria isolada da família? Quem? Qual seu banco principal?").

### 8. PWA-only, sem app nativo
**Hoje:** PWA via service worker inline. Funciona bem no iPhone, mas notificações push iOS são restritas, sem widgets de tela inicial.
**Comercial:** Considerar Capacitor (wrappers iOS/Android sem reescrever) ou React Native pra apps nativos. App Store/Play Store presence importam pra credibilidade.

### 9. Privacidade e LGPD
**Hoje:** Dados criptografados em trânsito (HTTPS) e em repouso (Firebase). Suficiente pra uso pessoal.
**Comercial:** Termos de uso, política de privacidade, encarregado de dados (DPO), processo formal de exclusão de conta, registro de operações de tratamento. Dados financeiros podem ser interpretados como sensíveis pela LGPD em alguns contextos.

### 10. Custos por usuário
**Hoje:** Firebase free tier cobre 1 usuário sobrando.
**Comercial:** Modelagem de custo por usuário ativo (Firestore reads/writes/storage, Auth, Hosting, eventual processamento server-side de PDF). Definir tier free vs pro.

### 11. Aliases de classificação (memória global por usuário)
**Hoje:** Os aliases aprendidos (descrição → categoria/subcategoria/grupo/identificação/isFixo) vivem no `state.aliasItems` do usuário. Quando Tiago atribui "DROGASIL 4940" como Casa·Saúde, todo usuário individual numa importação futura recebe a sugestão. A partir de v0.11.x, aliases também memorizam apelido livre (identificação) e flag de recorrência (isFixo).
**Comercial:** Aliases poderiam ter 3 camadas:
- **Globais (curadoria)**: "DROGASIL", "NETFLIX", "iFood" pré-cadastrados pela plataforma com categorias-padrão. Reduzem fricção do onboarding.
- **Por workspace**: aprendidos pela família.
- **Por usuário**: refinamentos pessoais.
Decisão de quando empurrar a sugestão global afeta a UX. Também: aliases globais geram dados agregados de comportamento de consumo — valioso comercialmente mas exige cuidado com privacidade/anonimização.

### 12. Parser PDF rodando no client
**Hoje:** PDF.js extrai texto no navegador do usuário. O PDF nunca sai do dispositivo.
**Comercial:** Excelente como diferencial de privacidade — pode virar argumento de marketing. Mas se quisermos suporte a mais bancos com parsers complexos (Nubank, BB, Caixa, C6, Bradesco, Santander, etc), processar server-side pode ser mais escalável (atualizar parsers sem republicar o app). Trade-off: privacidade vs manutenibilidade. Talvez híbrido: parsers comuns no client, fallback server-side opt-in.

### 13. Data de vencimento como data do gasto (não data da compra)
**Hoje:** Gastos de cartão entram no app com `data = data de vencimento da fatura`. A data da compra fica preservada em `dataCompra` apenas como flag visual.
**Comercial:** Decisão pragmática que funciona pra Tiago (faz contabilidade mental por mês de cobrança), mas pode confundir outros usuários que pensam por mês de uso. Apps comerciais geralmente oferecem opção: visualizar por data de compra OU por data de cobrança. Considerar toggle em Config.

### 14. Heurística de extração de chave do alias
**Hoje:** Primeira palavra significativa da descrição bruta, com fallback pra 2 palavras se a primeira for curta (≤4 chars). Remove prefixos comuns (IFD*, PG*, PD, DL*, etc.) hardcoded.
**Comercial:** Lista de prefixos é específica do mercado brasileiro e dos bancos cobertos. Internacionalização exigiria curadoria por região. Talvez ML simples (frequência de tokens) substitua heurística manual em escala.

### 15. Sem deduplicação automática de gastos importados (caso fatura)
**Hoje:** Se Tiago reimportar a mesma fatura de cartão, os gastos viram duplicados. Solução é manual (excluir os antigos).
**Comercial:** Detecção automática de duplicatas (mesmo cardId + valor + data ± 2 dias) com flag "já existe" na tela de revisão. Considerada e adiada pra Fatia 2.2 caso vire necessidade real. Pra comercial é mandatório — usuário leigo não vai entender por que tem 4 iFoods quando importou 2× sem querer.

### 16. Extrato bancário como "gap-filler", não fonte primária
**Hoje:** Etapa 5b ainda não implementada, mas a arquitetura está decidida: extrato vai entrar como **complemento** dos gastos manuais/importados de cartão, não como fonte total. Workflow desenhado para implementação:
- Usuário escolhe o período do extrato a importar. Default = "desde o último lançamento na conta + 1 dia" até a data do extrato.
- Parser sugere classificação por tipo de cada linha: entrada-renda, entrada-transferência, saída-consumo, saída-transferência, saída-pagamento-de-fatura, saída-tarifa.
- Tela de revisão flagra duplicatas contra `state.gastos` e `state.transferencias` da mesma `contaId`:
  - **Match estrito** (mesma conta + valor exato + data exata): "já cadastrado", desmarcado por default.
  - **Match aproximado** (mesma conta + valor exato + data ±2 dias): "possível duplicata", desmarcado por default, mostrando qual lançamento casou.
  - **Sem match**: marcado normal, pronto pra atribuir categoria.
- Aliases da Fatia 2.1 são reaproveitados: o que foi aprendido em fatura ajuda na classificação do extrato (e vice-versa).

**Comercial:** Isso é um **diferencial claro contra apps brasileiros típicos** (GuiaBolso, Mobills, Organizze) que importam tudo e geram duplicatas óbvias, deixando o usuário pra limpar manualmente. Argumento de marketing: "importe o extrato sem medo — só entra o que falta". O modelo de "gap-filler" também respeita melhor o usuário que registra manualmente em tempo real e não quer perder a sensação de controle.

**Pré-requisitos técnicos comerciais:**
- Dedup robusta é hard requirement (mais crítico aqui que no item 15, porque o caso de uso pressupõe co-existência de fontes). Em escala, pode aceitar variações: mesmo valor com 1 centavo de diferença (taxas residuais), descrições parecidas (fuzzy match).
- **Detecção de transferências pareadas** (saída em conta A + entrada em conta B mesmo dia mesmo valor = 1 transferência única, não 2 eventos) também é diferencial. Apps genéricos tratam como 2 lançamentos.
- **Detecção de pagamento de fatura** depende da Fatia 4 estar pronta (saída técnica vinda da `contaPagamentoId` do cartão). Por isso a ordem recomendada do roadmap é 4 → 5b.

### 17. Detecção de banco em fatura por marcadores múltiplos (logo é imagem rasterizada)
**Hoje:** Em fatura PDF do Itaú, "Itaú", "Personnalité" e CNPJ vêm como PNG embutido, não texto extraível pelo PDF.js. Detecção precisou cobrir marcadores secundários: nome do produto (`visa infinite`, `mastercard black`, que estão no corpo como texto) e telefones de atendimento (`4004 4828`, `0800 970 4828`). Cada banco novo (Sicoob, BTG, etc.) exige mapeamento manual desses marcadores.
**Comercial:** Trabalho linear por banco — pra escalar pra Nubank, BB, Caixa, C6, Bradesco, Santander seria necessário: (a) OCR fallback pra ler o logo quando texto falha, (b) catálogo server-side mantido pela plataforma de marcadores por banco, ou (c) cadastro guiado pelo próprio usuário ("qual seu banco?" antes de processar). Risco: banco muda layout ou marca, detecção quebra silenciosamente sem aviso. Em escala, monitoramento de "falhas de detecção" vira métrica de produto.

### 18. PDF.js fragmenta caracteres acentuados em items separados — normalização vira camada de infra
**Hoje:** Caracteres "ç", "ã", "é", "ó", "í" frequentemente vêm em items separados na extração. "Lançamentos" pode aparecer como 3 items: "Lan" + "ç" + "amentos". Quando agrupados por Y na reorganização, viram "Lan ç amentos" com espaços, e regex que depende deles quebra. Solução implementada: `normalizarFragmentacaoPDFJS(linha)` com regex que cola fragmentos em loop até estabilizar.
**Comercial:** Pra português brasileiro isso é universal — afeta qualquer parser de PDF processado via PDF.js. Em escala, normalização vira step obrigatório do pipeline de extração (não decisão caso-a-caso por parser de banco). Para outros idiomas com diacríticos (espanhol, francês, alemão), validar se a mesma fragmentação ocorre — provavelmente sim. Internacionalização passa por aqui.

### 19. Sistema de Y do PDF.js é nativo (origem inferior); PyMuPDF e outras libs server-side usam Y de tela (origem superior)
**Hoje:** Parsers que dependem de coordenadas Y (ex: detectar "Compras parceladas - próximas faturas" e cortar items abaixo desse marker) precisam saber em qual sistema estão. Hoje uso detecção adaptativa olhando item-âncora conhecido ("Resumo da fatura em R$"): se Y dele > mediana, sistema é PDF-native; senão, screen-native.
**Comercial:** Funciona desde que o item-âncora exista no documento. Bancos diferentes podem não ter "Resumo da fatura em R$" — cada formato exigiria mapeamento de âncora. Em escala: (a) catálogo de âncoras por banco mantido pela plataforma, (b) heurística independente de texto (ex: bounding box global da página), ou (c) abandono de PDF.js e migração pro server-side com biblioteca de Y previsível (PyMuPDF, pdfplumber). Última opção sacrifica o diferencial de privacidade do item 12.

### 20. Identificação (apelido 1-pra-1) vs grupo (agregador 1-pra-N) — escolha de schema com dois eixos
**Hoje:** Gasto pode ter `identificacao` (apelido livre tipo "Claude", "Netflix") OU `grupoId` (referência a um grupo cross-cutting tipo "Viagem Cancun"), ou ambos. Decisão explícita: não unificar — identificação discrimina 1 item dentro duma subcategoria; grupo agrega múltiplos gastos relacionados duma iniciativa.
**Comercial:** Terminologia pode confundir usuários leigos — "grupo" e "identificação" não são intuitivamente diferentes. Opções: (a) renomear identificação pra "apelido" ou "nickname", e grupo pra "projeto" ou "tag de projeto"; (b) unificar conceitualmente em "tag livre" e perder a granularidade do schema (perde-se a função de agregador automático do grupo); (c) explicar via onboarding com exemplos visuais. Trade-off entre simplicidade aparente e expressividade. Pra B2B (controle financeiro empresarial), a distinção tem valor — projetos vs centros de custo são conceitos contábeis distintos. Pra B2C, talvez unifique.

### 21. `isFixo: true` em gasto importado de fatura, sem `fixedExpense` espelho
**Hoje:** Implementei marca de fixo em gasto importado (assinaturas tipo Netflix, Claude, You Tube Premium) sem criar `fixedExpense` espelho — porque a fatura mensal já é a "engine" da recorrência, e criar fixedExpense duplicaria os lançamentos. Resultado: "fixo" agora tem 2 origens no sistema — replicação automática via fixedExpense ou flag direta no gasto. UI atual conta ambos juntos no filtro "fixos · X".
**Comercial:** Pra B2C funciona invisível — usuário não distingue origem. Pra B2B (especialmente controllers/contadores), pode precisar separar: "fixo recorrente automático" vs "fixo marcado manualmente" são auditoria-distintos. Pode virar 2 filtros separados, ou 1 filtro com sub-categorização visual. Schema preserva flexibilidade — campo `isFixo` é boolean simples, e `parentFixoId` identifica a origem (presente = clone de fixedExpense; ausente = marca direta). Decisão de UI fica pra revisão pré-comercial.

### 22. Descrição multi-linha em layout colunar — padrão Sicoob, pattern reutilizável
**Hoje:** Fatura Sicoob tem layout 4-col (DATA | DESCRIÇÃO | CIDADE | VALOR) com X fixos. Descrições longas (>~20 chars) quebram em 2-3 linhas físicas no PDF, e o PDF.js entrega cada linha como item em Y diferente. Lançamentos legítimos somem do parser genérico porque nenhuma linha sozinha matcha o regex `data + descrição + valor`. Solução implementada na Fatia 3.2: **âncora-de-DATA**. Cada lançamento é identificado pelo item na coluna DATA (X<50, texto "DD MMM"); todos os items dentro de |Y - Y_data| ≤ 5 pertencem ao mesmo lançamento (gap entre lançamentos é 18px); descrição multi-linha é concatenada ordenando os items da coluna DESCRIÇÃO por Y descendente. Funciona pra Sicoob, deve servir pra qualquer banco com layout colunar.
**Comercial:** Em escala (5-10+ bancos), virar **camada genérica do pipeline de extração**, não decisão caso-a-caso por parser. Detector de layout colunar via histograma de Xs (com freq ≥ 3), agrupamento por âncora-de-data com tolerância Y adaptativa (gap mediano + 1). O parser de cada banco fica responsável só por: (a) marcadores de detecção do banco, (b) filtros de descrições técnicas (anuidade, vantagem, juros, IOF), (c) regex de parcela X/Y caso varie de formato.

### 23. Floating-point edge case com Y exato na fronteira da tolerância
**Hoje:** Bug descoberto na Fatia 3.2: usar `Math.abs(it.y - anc.y) <= 4` (com gap de descrição multi-linha sendo exatamente 4) derrubava items legítimos por ruído de ponto flutuante (547.5000001 - 543.5 > 4 falsamente). Solução: sempre usar tolerância > gap medido (4 → 5).
**Comercial:** Risco silencioso — em testes com PDFs específicos pode passar, mas em outros falha. Em escala, o pipeline deve **medir o gap real por amostra** (mediana das distâncias entre items adjacentes em colunas com texto longo) e definir tolerância adaptativamente. Pra fail-soft: testes automáticos de regressão com auditoria de total bruto (item 26) pegam regressão imediatamente.

### 24. `parseValor` global do app não trata formato US com milhar
**Hoje:** A função `parseValor` (linha 5525 do index.html) é usada em todo o app pra interpretar valores monetários. Ela trata BR (1.234,56 e 50,00) e US-simples (50.00), mas falha em US com milhar (1,933.36): interpreta como "1.93336". Não causa bug no app hoje porque entradas manuais são sempre BR. **A fatura Sicoob usa formato US com milhar** — a Fatia 3.2 contornou implementando `parseValorSicoob` próprio no parserSicoob, sem mexer no global.
**Comercial:** Auditoria de débito técnico — em escala (com OFX, CSV de bancos internacionais, etc.) o `parseValor` global vai falhar silenciosamente. Refactor pendente: detectar formato pelo **último separador com 1-2 dígitos depois** (lógica já implementada em `parseValorSicoob`) e unificar. Risco de regressão moderado porque toca em todas as entradas monetárias do app — auditoria com testes de regressão antes do merge.

### 25. Anuidade do cartão como gasto vs encargo — decisão de produto opinionada
**Hoje:** Decisão da Fatia 3.2: anuidade entra como gasto (categoria a definir pelo usuário); vantagem progressiva (crédito redutor) e pagamento débito da fatura anterior ficam **de fora** da importação. Sicoob expõe anuidade parcelada (10/12, 11/12, etc.) — passa pelo aprendizado de alias como parcelamento normal e mês que vem auto-classifica.
**Comercial:** Pra B2C funciona — usuário não distingue se "anuidade" é consumo ou encargo, é só uma linha no gasto. Pra B2B (controllers/contadores), anuidade é fiscalmente "Despesas bancárias" / "Encargos com cartão", separada do consumo operacional. Pode precisar de categoria fiscal automática ("anuidade" → subcategoria "Encargos bancários" como sugestão de default). Esse tipo de regra-de-domínio cresce rápido; em escala, **biblioteca de regras de classificação fiscal** vira componente do produto B2B.

### 26. Auditoria de total bruto como gate obrigatório do pipeline de importação
**Hoje:** Convenção adotada nas Fatias 3.1 e 3.2: cada parser tem uma **"meta de auditoria"** — total dos lançamentos extraídos deve bater com o total bruto da fatura (extraído visualmente). Visa Infinite: 15 itens = R$ 12.972,27 (Fatia 3.1). Sicoob: 11 itens = R$ 1.947,92 (Fatia 3.2). Sem esse cross-check, regressões silenciosas passam (parser perdendo lançamento por um caractere a mais no regex, dupla-contagem de parcelas futuras, valor mal-parseado por 1 ordem de grandeza).
**Comercial:** Em escala (milhares de importações/dia em produção), vira **telemetria**: toda importação reporta `total_extraído / total_meta_estimado`. Discrepâncias >1% disparam alerta de "parser quebrou pra este banco/layout" — métrica de qualidade de produto. Também: na UI, exibir "✓ Total bate com a fatura: R$ X,XX" depois da importação aumenta confiança do usuário (especialmente em B2B onde auditabilidade é hard requirement). Hoje o app não mostra essa auditoria pro usuário; em escala comercial, virar feature visível.

### 27. Parser dual-mode (modo A com X/Y + modo B com texto bruto) — robustez genuína mas não foi a correção do bug Sicoob
**Hoje:** Durante debugging da Fatia 3.2 (15/maio/2026), achei que `pdfItems` chegava vazio em Safari iOS PWA (v0.12.0→v0.12.1), depois que modo A retornava lançamentos parciais (v0.12.2). **Ambos os diagnósticos estavam errados** — o `parserSicoob` nem foi chamado, porque o detector roteava pro Itaú genérico (ver item 29). v0.12.3 corrigiu o detector e o problema sumiu. **Apesar do diagnóstico errado, o dual-mode A/B e o best-of-N implementados continuam valendo como robustez genuína**: parser principal usa coordenadas X/Y (preferido, mais preciso, separa cidade da descrição); parser de backup usa textoBruto puro com âncoras "DD MMM" (mais robusto contra quirks de extração). Os dois modos rodam sempre, escolhe o de mais capturas, empate→A. Cobre cenários reais futuros: items chegando vazios por quirk de runtime, modo A perdendo casos específicos por sutil diferença de Y, ou layout diferente em uma fatura nova de outro banco.
**Comercial:** Em escala (milhares de devices, browsers, versões de SO), runtime quirks do PDF.js viram inevitáveis. **Pattern de "best-of-N" para parsers é replicável e simples**: cada modo independente, executa todos, escolhe o melhor por critério objetivo (contagem, soma vs total bruto extraído, etc.). Custo extra: ~milissegundos de CPU. Benefício: robustez sem precisar diagnosticar a causa raiz de cada quirk. Telemetria de "qual modo venceu por banco/SO" detecta regressões em produção. Em B2B isso vira SLA — "extração funciona em 99.5% dos PDFs" exige redundância.

### 27.1. Service Worker stale-while-revalidate vs deploy de patch
**Hoje:** Service Worker do app (linhas 10082-10122 do index.html) usa stale-while-revalidate com `ignoreSearch: true`. Resultado: query strings como `?v=12.3` NÃO forçam atualização. Quando um patch é deployado, usuário vê a versão antiga na PRIMEIRA visita pós-deploy (SW serve do cache, atualiza em background); a versão nova aparece SÓ NA SEGUNDA visita. Em ciclo de iteração rápida (4 patches em 1 sessão durante debug da Fatia 3.2), isso atrasa muito porque parece que "o fix não funcionou" quando na verdade só não foi servido ainda.
**Comercial:** Em B2C de uso casual (1x/semana), stale-while-revalidate é OK — usuário não percebe. Em B2C high-touch (uso diário) ou B2B (debugging com cliente), virar incidente. Soluções: (a) versionar `CACHE_NAME` em cada deploy (invalida cache imediatamente, mas força refetch completo); (b) detectar `?v=` no fetch handler e bypassar cache para essa request específica; (c) prompt "Nova versão disponível, recarregar?" mais agressivo (já existe mas o usuário pode ignorar). Decisão pré-comercial: balancear bandwidth vs imediatismo de updates.

### 28. Detecção de parcela X/Y em qualquer posição da descrição (não só no final)
**Hoje:** No modo B do parserSicoob (texto bruto sem coordenadas), a cidade fica colada na descrição depois da parcela ("IMUNOCENTRO SERV MED 01/05 BRASILIA"). Regex de parcela ancorada no final `/\s(\d{1,2})\/(\d{1,2})\s*$/` falhava nesse caso. Solução: regex global, valida atual/total (total>1, total≤99, atual≤total), pega o último match válido. Funciona pra parcela no fim (modo A) e no meio (modo B).
**Comercial:** Padrão recorrente em PDFs brasileiros — algumas operadoras (saúde, telefonia, faturamento parcelado) põem "01/05" antes do estabelecimento, antes da cidade, ou no meio do código de transação. Aplicar globalmente em parsers de extrato e fatura (não só Sicoob). Risco: falso-positivo se descrição naturalmente contém DD/MM válidos (improvável dentro do range de parcela). Cobertura de testes deve incluir descrições com números embutidos.

### 29. Ordem do detector de banco — emissor antes de bandeira/produto
**Hoje:** Bug encontrado nas v0.12.0, v0.12.1 e v0.12.2 (mascarado por hipóteses erradas durante 3 patches): detector verificava produto/bandeira ("Mastercard Black", "Visa Infinite") ANTES de verificar nome do emissor ("Sicoob", "BTG Pactual"). Faturas Sicoob com produto "Sicoob Mastercard Black Pro" eram roteadas pro Itaú genérico → parser genérico → 9 lançamentos em vez de 11, R$ 1.693,67 em vez de R$ 1.947,92. v0.12.3 inverteu a ordem: emissor primeiro (sinal único e forte), produto/bandeira só como fallback Itaú (justificado porque o logo Itaú é imagem rasterizada e não aparece no texto).
**Comercial:** **Hierarquia clara** quando o detector for crescer (mais bancos): primeiro emissor (banco), depois produto/bandeira. Bandeira NUNCA é evidência primária — só usar quando nenhum nome de emissor identifica. Documentar como regra interna. Em escala, virar tabela ranqueada por "força do sinal" (CNPJ > nome próprio do banco > telefone de atendimento > produto/bandeira > heurística genérica) que o detector percorre em ordem. Em B2B, expor essa lógica em telemetria/dashboard ajuda contadores a entenderem por que uma fatura foi roteada pra um parser específico.

### 30. Diagnóstico em camadas — checar a cadeia inteira antes de implementar correção
**Hoje:** Durante o debug da Fatia 3.2, gastei 2 patches (v0.12.1 e v0.12.2) implementando robustez no PARSER quando o bug estava no DETECTOR. Cada hipótese (pdfItems vazio → quirk de Y) era plausível e os testes locais confirmavam; mas em produção o parser nem era chamado. A pista que teria me poupado: o `__faturaDebug` veio "indisponível" — esse é um sinal forte de "este parser não rodou", não "este parser rodou mas falhou". Eu interpretei como "o objeto __faturaDebug não foi populado por algum motivo" (errado).
**Comercial:** Em escala, pipelines de importação têm 5+ etapas (upload → extração → detecção → parsing → revisão → confirmação → save). Cada etapa deve ter **debug observável separado**: "etapa X rodou, retornou Y". Sinal "indisponível" numa etapa específica deveria significar inequivocamente "essa etapa não rodou", e fazer pensar em "por que não chegou aqui?" — não "por que falhou aqui?". Em produção (B2B especialmente), telemetria deve capturar **breadcrumbs por etapa** com tempo, input/output, e razão de skip se aplicável. Acelera debugging em ordem de magnitude.

### 31. Layout colunar com ordem invertida no texto agrupado por Y — BTG vs Sicoob
**Hoje:** BTG e Sicoob têm a MESMA estrutura visual (3-4 colunas com X fixos: DATA | DESC | [CIDADE] | VALOR). Mas quando o PDF.js agrupa items por Y±2, o texto resultante sai em ordens diferentes: Sicoob mantém ordem visual ("DD ABR descrição cidade R$ valor"), BTG sai invertido ("R$ valor descrição DD Abr"). Razão: ordem dos items no content stream do PDF varia por gerador — quem o BTG usou produz items na ordem (X-direita, X-meio, X-esquerda) por linha, enquanto o Sicoob produz (X-esquerda, X-meio, X-direita). O modo A (com coordenadas X/Y) é imune porque usa âncora-de-DATA na coluna real X. O modo B (texto puro) precisa de regex específico pra cada ordem — implementei dois regexes no parserBTG (RX_INV pra BTG, RX_VIS pra padrão visual) e tento ambos em cada linha.
**Comercial:** Em escala, "ordem em que items aparecem no content stream do PDF" é decidida pelo software que gerou o PDF (não pelo PDF.js). Bancos terceirizam geração de fatura — pode mudar entre fornecedores ou versões. Pattern: **modo A com X/Y é a fonte da verdade**, modo B é fallback que precisa ser tolerante a ambas as ordens. Em produção, telemetria de "qual regex casou no modo B" ajuda a detectar geradores novos antes que quebrem.

### 32. Duplicação crescente de código entre parsers de banco — pendente consolidação
**Hoje:** Após Fatia 3.1 (Itaú Visa Infinite), 3.2 (Sicoob) e 3.3 (BTG), o app tem 3 parsers especializados + 1 genérico. Há duplicação significativa entre parserSicoob e parserBTG: parseValor robusto BR/US, extrairParcela em qualquer posição, montagem do objeto lanc, decisão best-of-A-or-B, escrita em window.__faturaDebug. Diferem só em: regex de mês (CASE vs Case), X-limites das colunas, filtros de descrição. Não consolidei agora pra não introduzir risco de regressão durante Fatia 3.3 (decisão consciente: estabilizar, depois refatorar). Está claro que com 4-5 bancos isso vira manutenção exponencial.
**Comercial:** **Refator pendente: `parserFaturaColunar(textoBruto, pdfItems, config)`** onde `config` define `{regexMesAncora, mesesMap, xCols: {data, desc, cidade?, valor}, filtroDesc, tratamentos especiais}`. Cada banco vira um objeto de config curto (10-15 linhas) em vez de função inteira (~150 linhas). Custo: 1 fatia técnica dedicada, sem nova feature visível. Benefício: adicionar um novo banco passa de "implementar parser inteiro" pra "fornecer config". Em B2B onde "suporte ao banco X" é frequente pedido de cliente, isso vira slope de entrega — diferença entre 1 dia e 4 horas de trabalho por banco.

### 33. Materialização + reconciliação: pattern híbrido pra projeção de gastos parcelados
**Hoje:** Fatia 4 ("Compromissos do mês") materializa parcelas faltantes ao importar uma compra parcelada (ex: IMUNOCENTRO 01/05 importada em MAI → cria 4 projeções 02/05, 03/05, 04/05, 05/05 nos meses seguintes com `isParcelaProjetada=true, pago=false`). Quando a fatura do mês seguinte chega com a parcela real, em vez de criar gasto novo, o app **reconcilia**: encontra a projeção compatível e ATUALIZA seus campos com dados reais (valor, data, categoria, marca pago=true, retira flag de projetada). Resultado: usuário vê os compromissos futuros antes mesmo das próximas faturas chegarem; não há duplicação. Pattern não-trivial — alternativas mais simples (só projeção visual, ou só materialização sem reconciliação) têm trade-offs piores: projeção pura não permite marcar pago manual; materialização pura duplica quando fatura nova é importada.
**Comercial:** Em escala, esse pattern habilita features de **planejamento financeiro real** (compromissos vs disponível, fluxo de caixa projetado, alertas de excedente) sem exigir double-entry bookkeeping. Risco: depende da chave de reconciliação ser estável (item 34). Em B2B, vira diferencial vs apps brasileiros que ou (a) só mostram a parcela do mês atual (sem visão futura) ou (b) duplicam gastos quando fatura nova chega. Refletir na UI: a tag "proj" explícita evita o pior cenário (usuário acha que já pagou algo que ainda é projeção).

### 34. Chave de reconciliação estável: `cardId | total | descricaoBruta normalizada`
**Hoje:** Pra reconciliar uma parcela importada de fatura nova com uma projeção pré-existente, o app gera uma chave determinística: `${cardId}|${parcela.total}|${descNormalizada}`. A `descNormalizada` é uppercase + remove acentos + remove não-alfanuméricos + slice(0,40). Critério-chave: usa `g.descricaoBruta` (texto ORIGINAL do PDF) em vez de `g.nome` (que o usuário pode ter editado). Razão: mesmo se o usuário renomear "IMUNOCENTRO SERV MED" pra "Plano Vacinas Família" no MAI, em JUN o PDF traz "IMUNOCENTRO SERV MED" → chave bate, reconcilia. Se usasse `g.nome` editado, perderia o match.
**Comercial:** Pattern reutilizável pra qualquer reconciliação de dados externos com edições do usuário: **preservar SEMPRE a fonte original** (`descricaoBruta`) separada da apresentação (`nome`). Em B2B/contabilidade, isso vira requisito de auditoria: "o que o sistema viu" vs "o que o usuário classificou" são campos distintos e ambos precisam ser preservados. Em escala, pode evoluir pra chave por fingerprint (descrição + valor exato + dia da compra) — mais robusta a colisões — mas com complexidade extra.

### 35. `state.schemaVersion` como marcador de migração one-time
**Hoje:** Fatia 4 introduziu 2 campos novos opcionais (`g.parcelaChave`, `g.isParcelaProjetada`, `g.isFixoProjetado`) sem migração disruptiva. Dados antigos continuam funcionando (campos undefined). Mas pra dados existentes terem o benefício novo (parcelas projetadas), a migração one-time `migrarParcelasV014()` roda automaticamente no boot, gerando `parcelaChave` em gastos existentes e materializando parcelas faltantes. Idempotente: marca `state.schemaVersion = '0.14'` quando termina; re-execuções skipam. Vantagem: zero coordenação manual ("rode esse script depois do deploy"); usuário só abre o app e está migrado.
**Comercial:** Pattern correto pra evolução de schema em apps client-side onde não há controle central de DB. Em escala: matriz de migrations `0.14 → 0.15 → 0.16 ...`, cada uma idempotente, rodadas em sequência baseado em `state.schemaVersion`. Equivalente a Rails migrations / Django migrations mas client-side. Em B2B com múltiplos clientes/contas, evita "1000 usuários em versões diferentes do schema". Risco a documentar: migrations precisam ser **defensivas** (gracefully handle dados malformados) — se uma migration crashar, o app inteiro fica preso.

### 36. Sincronização CRUD com projeções materializadas
**Hoje:** v0.14.1 adicionou o "compromisso de coerência" das projeções: quando o usuário pausa, edita ou remove um fixedExpense, as projeções futuras precisam refletir essa mudança imediatamente, MAS sem tocar nos gastos que já foram pagos (preservação de histórico). Implementado com duas funções: `limparProjecoesFuturasDoFixo(fixoId)` (filtra `parentFixoId === fixoId && pago === false && ym > corrente`) e `atualizarProjecoesFuturasDoFixo(fixoId)` (mesmo critério, mas atualiza campos em vez de remover). Cobertura: 5 cenários testados — pausar Netflix limpa 11 futuras mas mantém a paga; reativar recria; editar valor 30→45 atualiza só as futuras pendentes; remover limpa preservando histórico pago. **v0.14.2 estendeu o pattern pra gastos-parcela**: `limparProjecoesFuturasDeParcela(parcelaChave)` segue o mesmo critério (mesma chave + pago=false + ym>corrente). Disparada no handler de delete de gasto. UI: askConfirm avisa "esta compra parcelada tem N parcelas projetadas — também serão removidas" se houver projeções futuras pendentes. Toast pós-delete mostra o N. Cobertura adicional: 4 cenários testados.
**Comercial:** **Regra de ouro**: gastos pagos são FATOS — não devem ser alterados por mudanças no template (fixedExpense) ou por deleção de outra parcela. Gastos projetados são PLANOS — devem refletir mudanças no template e mudanças na compra parcelada. Critério único de discriminação: `g.pago === false && g.ym > currentYM()`. Pattern aplicável a qualquer sistema com "instanciação de templates" (recorrências, agendamentos, faturamento recorrente em SaaS B2B).

### 37. Saída técnica vinculada — pattern de "pagamento de fatura" como gasto reusando schema
**Hoje:** Etapa 4 (v0.15.0) fecha o ciclo cartão → conta: ao marcar fatura paga (manual ou automático via débito automático), o app cria um **"gasto técnico"** com `isPagamentoFatura: true`, `cardId: null` (pra não entrar no cálculo da própria fatura — evita loop), `contaId: card.contaPagamentoId` (debita a conta certa), `faturaCardId` + `faturaYm` (FK reversa pra recalcular). Vantagens do reuso de schema vs estrutura separada (`state.pagamentosFatura`): (a) `calcSaldoConta` já trata corretamente via filtro `pagamento !== 'cartao'`; (b) aparece no extrato da conta sem código novo; (c) save/load no Firestore funciona naturalmente; (d) renderização especial no extrato é simples (detectar a flag). Recalculo automático: ao adicionar/editar/remover gasto na fatura paga, `recalcularPagamentoFatura(cardId, ym)` ajusta o valor da saída técnica (ou remove se fatura ficar vazia). Boot: `processarPagamentosFaturaAutomaticos()` varre cartões com `debitoAutomatico=true` e cria pagamentos pra faturas vencidas (idempotente). Validação: 9/9 cenários.
**Comercial:** Pattern reutilizável: **"saída técnica vinculada"** pode modelar transferências espelho entre contas, ajustes com snapshot histórico, devoluções/estornos. Princípio: reusar o objeto-base do domínio (gasto/transação) com flags específicas + FK reversa permite que toda a infra existente (renderização, totais, saldos, sync) funcione sem código novo. Custo: ~10 linhas de código pra cada pattern novo. Em B2B (contabilidade, conciliação automática), esse pattern é a base da implementação. Atenção: `cardId: null` no pagamento técnico é crítico (sem isso entra em loop infinito no `calcularTotalFatura` — recálculo gerando recálculo). Princípio derivado: **fontes de cálculo devem ser separáveis das saídas do cálculo**.

### 38. SVG inline vs Chart.js para gráficos: decisão por especificidade visual
**Hoje:** Fatia 5.1 dos Relatórios (v0.16.0) introduziu 4 gráficos (total consolidado, por categoria com subcat empilhada, multi-mês com 2 curvas sobrepostas, small multiples 2x2). Apesar do Chart.js já estar carregado via CDN no head, optou-se por **SVG inline manual** em todos os 4. Razões: (a) controle pixel-perfect das duas linhas finas e discretas sobrepostas no mesmo viewBox (mediana tracejada `stroke-width=1` + média móvel suavizada `stroke-width=1.25` via Catmull-Rom→Bezier com tensão 0.18); (b) tooltip nativo via `title` attribute em vez de plugin do Chart.js; (c) cores de subcategoria derivadas em runtime via mix com branco (`_relMixWhite`, ratio 0..0.6) — abordagem inviável em Chart.js sem pré-calcular paleta; (d) zero JS de runtime extra no first paint (todo SVG sai pronto do render server-side equivalente); (e) drill-down via `data-rel-subcat` attribute resolve diretamente em event handler — sem custos do hit-test do Chart.js. Chart.js fica no head pra fatias futuras com gráficos mais complexos (pizza com tooltips ricos, mixed line+bar com legenda interativa).
**Comercial:** Princípio: **bibliotecas de chart genéricas existem pra resolver problemas genéricos**. Quando o requisito visual é específico (mediana + tendência sobrepostas + cores derivadas matematicamente + interação em segmento individual), o custo de domar a lib supera o de escrever SVG direto. Em B2B onde "exportação fiel pra PDF/print" e "tema customizável por cliente" são frequentes, SVG inline também é mais robusto (compatibilidade universal, fontes inheritáveis, vetorial). Risco a documentar: SVG inline não escala automaticamente em containers fluidos sem `viewBox` + `preserveAspectRatio` configurados corretamente — bug clássico que mata a UI mobile.

### 39. Hero sempre mostra o mês corrente, mesmo em modo multi-mês
**Hoje:** Fatia 5.1 dos Relatórios: o card "hero" no topo (total grande + pago + previsto + % comprometido em fixos) **sempre reflete o mês corrente selecionado**, mesmo quando o usuário seleciona "últimos 6 meses" no seletor de período. O seletor afeta apenas as seções abaixo (gráfico multi-mês, small multiples, lentes de grupos/nomes/identificações). Razão: confusão cognitiva. Se o hero virasse "média mensal" ou "total dos 6 meses", o usuário precisaria interpretar dois números no mesmo card (`R$ 50.100` "isso é o mês ou o período?"). Mantendo hero fixo no mês = âncora temporal estável, lentes abaixo = comparação histórica.
**Comercial:** Princípio aplicável a qualquer dashboard com seletor de período: **o número-âncora deve ter granularidade constante**. A informação adicional do período seleciona o que MUDA visualmente (gráficos, lentes), não o que serve de referência. Em B2B (analytics, BI), esse princípio reduz o "what does this number mean?" — disparador comum de tickets de suporte e UX mal-resolvido.

### 40. Lentes de relatório como seções empilhadas, não tabs
**Hoje:** Fatia 5.1: as 4 lentes de agrupamento do Relatório 1 (subcat, grupos cross-cutting, top nomes, top identificações) entram como **seções verticalmente empilhadas** num único scroll, não como tabs. Razão: tabs forçam escolha mutuamente exclusiva ("ver por subcat OU por grupo"), enquanto seções empilhadas convidam a comparação lateral ("o gasto em 'Plano Saúde' como subcat de Casa, também aparece no grupo 'Saúde Família' agregando Marina+Heitor"). O custo extra de scroll é tolerável em mobile pra um relatório que o usuário consulta com atenção (1-2 vezes por mês), não em fluxo rápido. Tabs ficariam melhores em views de consulta diária (entradas, contas), não relatórios analíticos.
**Comercial:** Princípio: **tabs escondem; seções empilhadas convidam comparação**. Em B2C de finanças pessoais, a história "uau, esse gasto aparece em 3 lentes diferentes" gera o momento de insight que vira retenção. Em B2B de relatórios, dashboards com lentes empilhadas tendem a ter NPS maior — mais informação visível sem clicks adicionais. Trade-off: scroll longo. Mitigação: seção de título com eyebrow forte (uppercase pequeno + ícone) cria âncoras visuais de scroll.

### 41. Visão econômica vs financeira: mesma base, lentes diferentes
**Hoje:** Fatia 5.2 (Relatório 2 — Fluxo de Caixa, v0.17.0) introduziu a **visão financeira** ao app: cartão de crédito não conta como saída no mês da compra; só conta quando a fatura é paga (`isPagamentoFatura: true`). Já o Relatório 1 (Fatia 5.1) usa **visão econômica**: cada compra de cartão conta no mês da compra, e pagamento de fatura é *excluído* do relatório (caso contrário duplicaria o consumo). Mesma base de dados (`state.gastos`), filtros opostos no runtime: `_relGastoConta` (econômica, exclui `isPagamentoFatura`) vs `_rel2FluxoConta` (financeira, exclui `pagamento === 'cartao'` mas inclui `isPagamentoFatura` que tem `pagamento: 'conta'`). O usuário escolhe a lente trocando de relatório via pills, não trocando de filtro dentro do mesmo relatório.
**Comercial:** Pattern aplicável a qualquer sistema com múltiplas perspectivas sobre os mesmos fatos: **schema único, lentes múltiplas via funções puras de runtime**. Em B2B de contabilidade isso é fundamental — regime de caixa vs regime de competência (accrual basis) é exatamente essa dicotomia, e cada empresa precisa dos dois pra fechamento fiscal e gestão financeira respectivamente. Em fintechs, "cartão como compra" vs "cartão como crédito" também é a mesma estrutura. Princípio derivado: **se duas visões disputam o mesmo render, separe a UI; se compartilham infra mas precisam de filtros opostos, mantenha o filtro como parâmetro explícito do cálculo, não como branch no render**. Custo de carregar 2 relatórios separados (5.1 e 5.2): zero esforço pra fatia 5.3 e além — só adicionar pill + sub-render.

### 42. Dispatcher com pills resolve "vista com sub-vistas"
**Hoje:** Antes da Fatia 5.2, `renderRelatorios()` era um render monolítico do Relatório 1. Pra adicionar o Relatório 2 (e os futuros 5.3, 5.4) refatorou-se em **dispatcher leve**: renderiza pills no topo + delega pra sub-renders (`_renderRel1Inner()` e `_renderRel2Inner()`). Cada sub-vista tem estado próprio (`relState`, `rel2State`) que é preservado ao alternar — usuário pode trocar entre Rel 1 e Rel 2 sem perder ym/periodo/toggle de cada um. Anti-pattern alternativo (rejeitado): cobrir tudo num único render gigante com condicionais (`if (currentReport === 'rel1') ... else ...`). Em 3-4 fatias isso vira inmanutenível.
**Comercial:** Pattern aplicável a qualquer view que ganha múltiplos modos depois da implementação inicial: refatorar pro **dispatcher + sub-renders independentes** custa ~30 linhas e paga dividendos imediatos. Em B2B de SaaS isso é o padrão "tabs com state preservado" — esperado pelo usuário. Princípio: **estado independente por sub-vista é o que faz tabs serem úteis em vez de irritantes**. Se ao trocar de tab tudo reseta, o usuário precisa reconfigurar — atrito que mata o uso da feature.

### 43. Schema asymmetry como microcopy honesta
**Hoje:** Fatia 5.2 expôs uma assimetria do schema atual: `state.gastos` tem `categoria` (member-bound: 'casa', 'marina', ..., 'tiago'), mas `state.entradas` tem só `{id, fonteId, contaId, valor, data, ym}` — sem campo de membro. Implicação: o toggle "Incluir Tiago no fluxo da família" não pode afetar entradas simetricamente. Decisão: em vez de fingir simetria ou inventar lógica frágil ("considera entradas via conta Tiago"? "considera entradas com observação contendo 'tiago'"?), escrever **microcopy explícita**: quando o toggle está ON, aparece "*entradas: família · saídas + invest: família + Tiago*". Quando OFF, "*entradas e saídas: apenas família*".
**Comercial:** Princípio: **quando o modelo de dados tem cantos não-ortogonais, explicitar a assimetria via texto é mais robusto do que tentar resolver visualmente**. Usuário informado é usuário confiante; usuário enganado por simetria fake é usuário que perde confiança ao descobrir. Em B2B (financeiro, contabilidade), microcopy honesta sobre o que está sendo calculado é diferencial — bons relatórios financeiros sempre têm rodapé técnico ("inclui/exclui X, Y, Z"). Refator de schema (adicionar `fonteOwner` nas entradas?) fica como decisão consciente futura, não dívida silenciosa. NB pessoal pro Tiago: se algum dia a Juliana entrar no app com renda própria registrada (Etapa B futura), aí o `fonteOwner` vira necessário. Hoje, simplificação assumida.

### 44. Coerência visual: "tudo colunas, nada linhas" como princípio estético
**Hoje:** No primeiro draft da Fatia 5.2 (v0.17.0), o multi-mês do Rel 2 usava barras agrupadas (E·S·I) com **linha tracejada Bezier de saldo sobreposta** — mistura de barras + linha que parece poderosa em teoria mas vira ruidosa visualmente em mobile pequeno (linha luta com as barras pela atenção). Nos ajustes da v0.18.0, a linha foi substituída por uma **4ª coluna divergente** por mês: pra cima se saldo positivo (verde), pra baixo se saldo negativo (vermelho), com eixo zero como linha cinza sutil. Resultado: gráfico mais legível com mesmo conteúdo informacional. Trade-off documentado: comparações de evolução temporal são mais difíceis sem linha (olho perdeu a "trajetória"), mas o ganho de clareza local supera em multi-mês curto (3-6 meses).
**Comercial:** Princípio: **dentro de um mesmo relatório, escolher um vocabulário visual e mantê-lo**. Misturar barras + linhas + areas no mesmo gráfico mobile não funciona — desktop tolera mais, mobile pune. Em dashboards B2B, "consistência > variedade" é regra de ouro pra retenção de atenção. Em mobile específico: prefira sempre colunas pra valores discretos (mês, categoria, item) e reserve linhas pra séries contínuas longas (>12 pontos) onde a tendência é a história. Lição prática: o primeiro draft visual quase sempre tenta "fazer demais" — o segundo draft simplifica e fica melhor.

### 45. Subcategorias consolidadas como lente complementar de "consumo total"
**Hoje:** Os ajustes da v0.18.0 adicionaram no Rel 1 uma seção nova: **"consumo · por subcategoria (top 10)"** — agrega todas as subcategorias somando todas as categorias incluídas (ex: Moradia da Casa + Marina + Heitor + Juliana → R$ X). Visualmente colunas verticais finas em scroll horizontal, cor única terracota (`--flow-out`), valor total acima em Fraunces, nome embaixo. Drill-down: lista todos os gastos daquela subcat em qualquer categoria do período. Complementa o gráfico "por categoria · subcategorias" (que mostra subcat × categoria) — esse novo mostra apenas subcategoria. Decisão de tomar a tabela cat × subcat (2D) e projetar em apenas 1 das 2 dimensões — porque na prática só uma das duas dimensões geralmente importa de cada vez.
**Comercial:** Princípio: **dados multidimensionais merecem múltiplas projeções, não uma matriz gigante**. Em B2C de finanças pessoais, a pergunta "quanto gastei com transporte no total" é muito comum, mas a estrutura cat × subcat força a resposta "transporte da Marina + transporte do Heitor + ...". Lente consolidada resolve isso com uma única coluna — UX que reduz o trabalho cognitivo do usuário. Em B2B (analytics), oferecer pivot dinâmico via dropdown ("agrupar por: categoria | subcategoria | conta | etc") é o equivalente. Custo da lente nova: ~50 linhas de código (reusa toda a infra de drill-down).

### 46. Escala honesta vs escala adaptativa em gráficos com magnitudes diferentes
**Hoje:** A seção "composição do mês" no Rel 2 (v0.18.0) mostra 3 colunas: Entradas (R$ 9k), Saídas (R$ 7k), Saldo (R$ 2k). Decisão de design: **mesma escala vertical pras 3 colunas**. Consequência: a coluna de saldo aparece como ~22% da altura de entradas, comunicando visualmente "saldo é uma pequena fração do fluxo total". Alternativa rejeitada: escala adaptativa por coluna (cada uma usa 100% da altura disponível) — daria visualmente "saldo grande" mesmo quando R$ 2k é proporcionalmente pequeno frente a R$ 9k. Saldo positivo aparece coluna pra cima em verde; negativo, pra baixo em vermelho.
**Comercial:** Princípio: **gráficos devem comunicar magnitude relativa de forma honesta, mesmo quando isso torna alguns elementos pequenos**. Em B2B (relatórios financeiros), escalas adaptativas são fonte de confusão e desconfiança — um saldo de R$ 2k aparecer "grande" porque a escala foi ajustada faz o usuário sub-estimar a importância de R$ 7k de saídas. Em B2C (apps de finanças), o mesmo: "meu saldo está enorme" é uma ilusão que vira surpresa negativa no fim do mês. Trade-off: legibilidade local (coluna pequena pode ter texto cortado) vs honestidade global (proporções fiéis). Mitigação: valor em texto sempre visível acima da coluna (em Fraunces), garantindo que mesmo a barra mais fina comunique seu valor exato.

### 47. Parametrização preventiva paga dividendos compostos
**Hoje:** Na Fatia 5.1 (v0.16.0), todas as funções de cálculo (`relCalcHero`, `relCalcTotalConsolidado`, `relCalcGrupos`, `relCalcTopNomes`, etc.) foram escritas aceitando `cats` (array de categorias) como parâmetro, mesmo que naquela fatia o array sempre vinha de `relCategoriasIncluidas()`. Custo extra na hora: ~0 linhas — apenas mudar `g.categoria === fixedKey` por `cats.indexOf(g.categoria) >= 0` no filtro. Benefício acumulado: na Fatia 5.2 (Rel 2) essas funções já receberam um cats diferente (categorias de saída); na Fatia 5.3 (Rel 3 = Meus Gastos · Tiago, v0.19.0), 80% dos cálculos foram reusados literalmente com `['tiago']` ou `['casa', 'marina', 'heitor', 'juliana']` sem nenhum código novo. A Fatia 5.3 inteira saiu em ~611 linhas (vs 1.077 da 5.1) — quase 50% de redução de esforço, com mais funcionalidade (toggle comparativo).
**Comercial:** Princípio: **se uma função vai ser usada em mais de 1 contexto previsivelmente, parametrize o contexto desde a primeira chamada**, mesmo que pareça "premature parametrization". O ROI vem composto: cada nova fatia paga uma fração do investimento original, mas o investimento original foi quase zero. Em B2B (SaaS de finanças/relatórios), esse pattern é diferencial competitivo — features novas saem rápido porque a infra já estava pronta. Anti-pattern oposto: hardcoding por suposição de "YAGNI estrita" (you aren't gonna need it) — funciona pra coisas isoladas, mas não pra módulos com claro potencial de variação. Heurística: **se você consegue imaginar 2+ contextos onde a função pode ser usada, parametrize**. Se imagina 1 só, não parametrize.

### 48. Toggle muda enquadramento, não disponibilidade
**Hoje:** No Rel 3 (Meus Gastos · Tiago, v0.19.0), o toggle "Comparar com a família" não esconde nem revela informação — ele troca a perspectiva sobre os mesmos dados. Quando OFF: hero focado no Tiago com cor verde, multi-mês com barras stacked + curvas (mediana + média móvel). Quando ON: hero com 2 colunas (Tiago + Família, %), multi-mês com barras agrupadas Tiago vs Família. Anti-pattern rejeitado: toggle "mostrar gráfico comparativo: sim/não" — vira interruptor binário que parece útil mas atrapalha o fluxo de análise (o usuário acaba precisando ligar/desligar várias vezes pra navegar). Pattern correto: toggle como **dimensão analítica** — solo vs comparativo são duas perspectivas válidas do mesmo dado, escolhidas conforme a pergunta sendo feita ("quanto gastei?" vs "como me comparo?").
**Comercial:** Princípio: **em UI analítica, toggles devem reformular a apresentação, não controlar visibilidade**. Em B2B (BI, analytics), dashboards bons usam toggles pra alternar entre "valores absolutos" e "% do total", entre "diário" e "acumulado", entre "valor" e "variação vs período anterior" — sempre mostrando os mesmos KPIs sob lentes diferentes. Em B2C (apps de finanças/saúde), o mesmo padrão: "meta diária" vs "média semanal", "saldo atual" vs "projeção fim do mês". Princípio derivado: **toggle que esconde informação obriga o usuário a memorizar o estado anterior pra comparar — quebra a análise**. Toggle que reformula deixa a comparação no próprio gesto de alternar.

### 49. Estruturas paralelas não-sincronizadas = bug latente esperando manifestar
**Hoje:** O bug do HANSARD (v0.19.1) veio de duas estruturas que representavam "a mesma verdade" mas evoluíram separadamente: `state.investMods` (modalidades com UI completa em Config) e `state.subcatsInvestimento` (lista consultada pelo filtro `isInvest()`, sem UI, com default fixo). O usuário cadastrava modalidades pela UI esperando que isso "ativasse o reconhecimento", mas o filtro nunca lia essa lista. Resultado: HANSARD ficava com `g.isInvest = false` na criação e o app nunca o reconhecia retroativamente. Correção foi tripla: (1) `isInvest()` consulta as duas listas (consolidação no leitor), (2) `_isInvestRuntime(g)` aplicado em 17 lugares ao invés de `g.isInvest` direto (corrige retroativo sem migração obrigatória), (3) migration silenciosa idempotente no boot pra reconciliar gastos antigos.
**Comercial:** Princípio: **toda vez que existirem duas estruturas representando a mesma verdade, é questão de tempo até dessincronizarem**. Em B2B (SaaS), isso é fonte clássica de bugs em integrações (CRM + email marketing, sistema fiscal + ERP, contabilidade + bancos). Soluções: (1) ponto único de consulta com fan-out (um "leitor universal" como `_isInvestRuntime`), (2) sincronização event-driven com idempotência (mudou A, atualiza B automaticamente), (3) migration silenciosa idempotente no boot. Lição prática derivada: **se você criou uma UI pra editar uma estrutura, garanta que o filtro/leitor consulta essa estrutura — nunca uma "lista paralela" hardcoded ou com default fixo**. Em produtos vendidos, isso é diferencial: bugs de sincronização são extremamente difíceis de reproduzir pelo cliente e geram fricção crônica.

### 50. Regimes contábeis (econômico vs financeiro) precisam de microcopy constante
**Hoje:** O Tiago questionou repetidamente "como gastei R$ 56k mas saiu só R$ 43k da minha conta?" — confusão entre regime de competência (Rel 1: gasto conta no mês em que aconteceu) e regime de caixa (Rel 2: só conta o que saiu da conta no mês). Tecnicamente: cartão comprado em maio fica no Rel 1 de maio, mas só aparece no Rel 2 quando a fatura é paga (junho). Mesmo entendendo intelectualmente uma vez, o usuário esquece — porque na vida normal "gasto" e "saída de dinheiro" coincidem. Solução implementada: microcopy em itálico discreto **em cada hero** dos relatórios. Rel 1: "regime econômico: gastos contam no mês em que aconteceram". Rel 2: "regime de caixa: compras de cartão entram no mês em que a fatura é paga. Faturas não pagas ainda não saíram." Recorrente, em todo render, sem precisar abrir FAQ.
**Comercial:** Princípio: **diferenças conceituais sutis precisam de lembrete CONSTANTE no contexto onde aparecem, não numa documentação separada**. Em B2B (contabilidade, finanças, BI), termos técnicos precisam de tooltips e microcopy *in situ* — não num glossário. Em fintechs B2C ("rendeu X% no mês mas tô vendo Y% no extrato — diferença?"), a comunicação clara de regime é o que separa apps que confundem usuários dos que educam. Diferencial concreto: cada KPI tem 1 linha explicativa sob ele, em itálico, com peso visual reduzido — visível mas não-intrusiva. **Microcopy bem feita é UX educativa contínua — diferente de onboarding pontual que o usuário esquece**.

### 51. Hierarquia de fallback escala melhor que dimensões paralelas
**Hoje:** Pra "tipos de investimento" (v0.21.0), o Tiago tinha 2 opções: criar uma subcategoria por tipo (HANSARD, CDB Itaú, FII XPLG...) — vira poluição cadastral quando ele só faz 1 aporte por mês em cada — OU usar campos genéricos (identificação, grupo) pra diferenciar. Solução escolhida: **hierarquia de fallback** — `_relInvestChaveTipo(g)` retorna agrupamento por `identificacao` se preenchida, senão por nome do `grupo`, senão por `subcategoria`. Auto-degradação elegante: mesmo sem ninguém preencher nada extra, aparece agrupado pela subcat. Quando preenche identificação, granularidade aumenta. Quando preenche grupo, vira intermediário. Tag visual (`•` idf, `🔗` grupo, `—` subcat) indica origem do agrupamento — usuário entende como sua escolha está sendo lida.
**Comercial:** Princípio: **quando o usuário tem N formas de preencher a mesma informação (sinônimos no schema), defina uma hierarquia explícita e mostre visualmente qual está em uso**. Anti-pattern: forçar uma forma única (rígido) ou tratar todas como independentes (sobreposição confusa). Pattern correto: hierarquia + transparência visual. Em B2B (CRM, helpdesk), isso aparece como "preencha categoria OU tag OU ambos — o sistema agrupa pela mais específica que existir". Em fintechs B2C: "classifique a transação OU categorize OU adicione comentário — relatórios usam o que estiver mais detalhado". Lição operacional pra produtos: oferecer múltiplas formas de preencher tem custo zero de UX **se houver hierarquia clara e indicador visual**. Sem isso, usuário fica indeciso ("uso qual?") e dados ficam inconsistentes.

### 52. Refactor sistêmico requer enumeração explícita de callers
**Hoje:** Na v0.20.0, modifiquei o filtro `_relGastoConta` pra criar `_relGastoSaida` (inclui invest no total). Apliquei nos relatórios mas esqueci do módulo Família/Tiago/Config, que usa outra função (`totaisCategoriaMes`). Tiago descobriu na v0.20.2: "os investimentos continuam fora da soma total de gastos da família no módulo 3". Fix exigiu nova rodada (estender `totaisCategoriaMes` retornando invest, atualizar 3 callers). 30 segundos de `grep totaisCategoriaMes` no início teriam economizado uma versão inteira.
**Comercial:** Princípio: **antes de modificar qualquer função "central" (chamada de mais de 1 lugar), enumere os callers explicitamente e raciocine sobre cada um**. Em B2B (SaaS com módulos), isso é diferença entre updates seamless e updates que quebram features esquecidas. Em codebases médios/grandes, ferramentas estáticas (grep, IDE find usages, dependency graph) tornam isso barato — mas a disciplina de USAR essas ferramentas antes do refactor é o que separa engenheiros sêniores de juniors. Lição operacional pra agentes que escrevem código (humano ou IA): **antes de mudar API de função compartilhada, listar os callers + raciocinar sobre cada um na resposta**, mesmo que pareça "óbvio que só afeta X".

## Diferenciação real do produto

Manter destacado o que pode virar argumento comercial:

- **Família com membro isolado** — útil pra mães solo, casais com finanças parcialmente separadas, casas com pais idosos.
- **Eixos cross-cutting (grupo + subcategoria + identificação)** — único entre apps BR. Ex: rastrear todo gasto com "Honda Civic" cruzando IPVA + gasolina + seguro + manutenção, mesmo estando em subcategorias diferentes. Identificação adiciona granularidade 1-pra-1 pra streamings e assinaturas.
- **Parser nativo de fatura BR** (Fatia 3) — Itaú Black, Itaú Infinite, Sicoob, BTG são cartões premium do segmento high-income. Qualidade de extração vence apps genéricos.
- **Parser Itaú Visa Infinite robusto** (Fatia 3.1) — diferencial técnico não-trivial de replicar: layout 2-col com histograma de Xs, normalização de fragmentação de PDF.js, detecção adaptativa de sentido Y, corte ancorado da seção "Compras parceladas - próximas faturas" com seleção do melhor candidato por posição. 5 iterações de fix em validação real. Reflete investimento de engenharia que apps genéricos não fazem.
- **Parser Sicoob robusto com descrição multi-linha** (Fatia 3.2) — captura corretamente lançamentos que quebram em 2-3 linhas físicas (descrições longas com parcela X/Y no meio, ex: "IMUNOCENTRO SERV MED 01/05"). Usa âncora-de-DATA + agrupamento por Y adaptativo. Parser genérico perderia esses lançamentos silenciosamente — apps comerciais provavelmente perdem. Auditoria de total bruto pegou e validou (R$ 1.947,92 bate centavo a centavo com a meta).
- **Parser BTG Pactual** (Fatia 3.3) — terceiro banco com parser dedicado. Layout 3-col em case-mista ("Abr" não "ABR") com ordem invertida no texto agrupado por Y. Confirmou que o pattern dual-mode (X/Y + texto bruto fallback) é replicável: parser BTG implementado em ~150 linhas, validado em 3 cenários (items OK, vazios, null) — todos R$ 40,90 centavo a centavo. Próximo banco já cai num framework conhecido.
- **Compromissos do Mês** (Fatia 4) — visão consolidada de gastos fixos + parcelas projetadas, com 3 perspectivas (Família · Tiago · Total) e toggle pra incluir parcelas. Diferencial técnico: ao importar uma compra parcelada, o app materializa as parcelas faltantes nos meses seguintes (`isParcelaProjetada=true, pago=false`); quando a próxima fatura chega, ele **reconcilia** automaticamente em vez de duplicar. Usuário vê 12 meses de fixos projetados + parcelas projetadas até o fim do parcelamento. Em B2C, vira "vou ter R$ X comprometido em JUN" — visibilidade que apps genéricos não oferecem.
- **Relatório 1 — Gastos & Saídas em 4 lentes simultâneas** (Fatia 5.1, v0.16.0) — único entre apps BR a oferecer 4 modos de agrupamento (subcategoria, grupo, nome, identificação) num **mesmo relatório com drill-down universal**. Cada lente é uma seção empilhada (não tab) — apps brasileiros tipicamente forçam escolha por dropdown ou tabs. Pago + previsto separados visualmente (faixa tracejada pra previsto). Modo multi-mês com pilha por categoria + 2 curvas finas sobrepostas (mediana tracejada + média móvel suavizada via Bezier) + small multiples por categoria com tendência individual (↑ % / ↓ % / → estável). Grupos cross-cutting marcados com ✦ quando aparecem em mais de uma categoria. Toggle inline pra incluir Tiago como mais uma categoria — preserva o princípio "Tiago é isolado" sem perder a opção de visualizar o consolidado total.
- **Relatório 2 — Fluxo de Caixa em visão FINANCEIRA** (Fatia 5.2, v0.17.0) — único entre apps BR a separar com clareza **regime de competência** (Rel 1: cada gasto conta no mês da compra) de **regime de caixa** (Rel 2: cartão só conta quando a fatura é paga). Apps brasileiros típicos confundem os dois ou só oferecem o regime de competência. Hero de 3 colunas (Entradas verde · Saídas terracota · Investimentos navy) com mini-trend ("↑ X% vs mediana 6m") e saldo do mês (verde se positivo, terracota se negativo). Modo multi-mês com barras agrupadas (3 lado a lado por mês) + linha de saldo tracejada sobreposta. 4 lentes (entradas por fonte · saídas por categoria · investimentos por subcat · saídas por conta de origem) com drill-down universal — drill-down de conta mostra "qual conta sangrou mais este período" útil em multi-conta brasileiro (Itaú + Sicoob + BTG). Pagamento de fatura aparece naturalmente como saída no mês do débito automático — visualização honesta do fluxo real do caixa do mês.
- **Relatório 3 — Meus Gastos com comparativo opcional** (Fatia 5.3, v0.19.0) — pattern de "vista isolada com comparativo opt-in via toggle". Tiago tem sua vista própria (cor verde, gastos próprios), mas pode ativar "Comparar com a família" pra ver lado a lado Tiago vs Família com percentuais, sem perder o foco do Tiago. Diferencial cognitivo: o toggle muda **enquadramento** sobre os mesmos dados, não controla visibilidade — solo vs comparativo são duas perspectivas válidas. Small multiples adaptados: em vez de mini-gráficos por membro (como no Rel 1), são mini-gráficos por subcategoria do Tiago (top 4) com mediana + tendência individual. Aplicação B2B clara: relatórios de centro de custo onde uma área específica quer se comparar com o resto da empresa sem perder o foco da própria operação.

- **Regimes contábeis explícitos e visualmente coerentes** (sessão 17/maio, v0.20.x) — o Finança trata regimes contábeis como cidadãos de primeira classe: Rel 1 é regime econômico (gasto conta no mês em que aconteceu, mesmo que pago depois) e Rel 2 é regime de caixa (só conta o que saiu da conta no mês). A diferença é técnica mas frequentemente confunde usuários — apps brasileiros típicos misturam os dois ou só oferecem o regime econômico. Solução implementada: microcopy em itálico discreto em cada hero ("regime econômico: gastos contam no mês em que aconteceram" / "regime de caixa: compras de cartão entram no mês em que a fatura é paga"). Recorrente no contexto onde aparece, não num FAQ separado. Pra finanças pessoais, isso é UX educativa contínua. Pra B2B (contabilidade/BI), é o tipo de detalhe que separa "ferramenta confusa que precisa de treinamento" de "ferramenta que ensina enquanto é usada".

- **Compromissos do mês como visão estratégica complementar** (Fatia 5.3-extras, v0.21.x) — pra além do "quanto gastei", o Rel 1 do Finança responde "quanto do mês já está pré-comprometido com recorrências fixas e parcelas de cartão em curso". Visualização: 1 coluna stacked grande (parcelas em mostarda escura em cima, fixos em terracota embaixo) com legenda lateral discriminando os valores. Cores **bem distintas** comunicam que são naturezas diferentes (fixos = recorrência indefinida, parcelas = compromisso finito até quitar). Drill-down em cada item. Diferencial: a maioria dos apps brasileiros não separa "parcelas em curso" de "gastos do mês" — confunde o usuário sobre qual é o "mínimo inegociável" do mês. Aplicação B2B: "compromissos firmes" no fluxo de caixa empresarial — folha + aluguel + financiamentos vs operação variável.

- **Investimentos hierárquicos com fallback elegante** (Fatia 5.3-extras, v0.21.0) — pra evitar poluição cadastral (criar 1 subcategoria por tipo de investimento quando o usuário faz 1 aporte por mês em cada), o Finança permite cadastrar **uma modalidade só** ("Investimento") em Config e diferenciar tipos via identificação ou grupo (livres por lançamento). Os relatórios agrupam por: identificação se preenchida → senão grupo → senão subcategoria. Tag visual (`•` `🔗` `—`) indica origem do agrupamento. Auto-degradação elegante: funciona mesmo se ninguém preencher nada extra. Aplicação B2B: classificação flexível de transações (CRM/ERP/contabilidade) sem forçar taxonomia rígida — o usuário preenche o que faz sentido na hora, sistema agrupa pela informação mais específica disponível.
- **Privacidade por padrão** — saldos escondidos, dados isolados, "tap to reveal" estilo private banking. PDF nunca sai do dispositivo.
- **Memória de classificação automática** (Fatia 2.1) — aprende com cada atribuição. Atrito mensal cai a quase zero depois de 2-3 importações.
- **Re-aprendizado em edições** (v0.9.6) — corrigir manualmente um gasto também atualiza o alias correspondente. Sistema fica mais inteligente conforme o uso.
- **Alias estendido com identificação e marca de fixo** (Fatia 3.1.5) — sistema não só aprende "Netflix é Lazer/Entretenimento", aprende também "Netflix é recorrente" e "Netflix é o apelido que usuário usa". Refinamento progressivo automático que apps genéricos não oferecem.
- **Totais consolidados unificados** (v0.9.7) — uma única visão que agrega gastos por grupo (quando há) ou por nome (quando não há). Resolve o caso "todos os iFoods juntos" naturalmente.
- **Extrato como gap-filler** (Etapa 5b, item 16) — importar sem medo de duplicata. Apps genéricos não fazem isso.
- **Auditoria de total bruto** (item 26) — em escala comercial, pode virar feature visível ("✓ R$ X,XX confere com a fatura") + telemetria de qualidade de parser por banco.
